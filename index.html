X-macro

发表于 2026-03-27 阅读次数： Valine：

0. 背景

使用 C 语言在对于一个特定的硬件对象/实例进行驱动实现时，这个对象/实例往往存在一系列可以外部调用的动作接口，我偏向于的设计是，构建一个结构体，然后将这个对象/实例的所有可以外部调用的动作接口（函数指针）包含其中。那么在后续开发过程中，往往只需要查看这个结构体，便可快速回顾到这个对象涉及的接口调用，十分方便快捷，并且易于管理。例如在 esp-idf 的 spiffs 文件系统的调用接口：

但是在一些场景下，上述方法存在一些局限性。

局限性一

安全编码的要求比较高时，函数指针被部分安全标准（如 MISRA C、CERT C）谨慎对待。

风险 说明
控制流不透明 静态分析工具难以追踪实际调用目标
间接跳转攻击 函数指针是 ROP/JOP 攻击的常见目标
未定义行为 类型不匹配、空指针调用等难以检测
可读性下降 复杂声明降低代码可审计性

虽然，上述风险可通过：

  • 用 typedef 定义清晰类型；
  • 放入 static const 表中，运行时不可变；
  • 调用前检查 NULL；
  • 索引访问时做边界检查；
  • 绝不从外部数据构造函数指针；
  • 编译时开启控制流保护；
  • 在代码注释中说明为什么需要函数指针

得到尽可能的解决，但编写一个函数指针需进行上述一系列的 check，工作量激增，且在后续维护过程中徒增烦恼。

局限性二

在涉及类似 FreeRTOS 移植运行于资源紧张的微控制器场景下，任务栈的空间分配需尽可能“恰到好处”。这时，往往通过查询编译生成的 .htm 文件，获取函数调用链及调用深度，并附加中断发生时所需的栈帧空间去确定“恰到好处”的任务栈大小。但是，若函数执行过程涉及到通过函数指针去跳转进行函数执行，往往会导致过程中涉及的堆栈情况无法有效统计，进而导致无法设计“恰到好处”的任务栈大小。

1. X-macro

在 “将一个对象/实例的所有可以外部调用的动作接口（函数指针）包含在一处” 目的前提下，并且规避上述 局限性一 及 局限性二 的最直接办法是：将一个对象/实例的所有可以外部调用的动作接口（函数）写到一个函数中，内部通过 switch-case 进行不同动作接口的跳转。
